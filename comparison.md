2026년 상반기에 세 회사가 비슷한 시기에 "사내 데이터 분석 에이전트를 어떻게 만들었는가"를 공개했습니다. 같은 문제(자연어 질문 → 정확한 분석)를 풀지만 **정답의 출처를 무엇으로 보느냐**에서 접근이 갈립니다.

## 1. 각 회사의 접근법

**OpenAI — "의미는 코드에 있다" (자동 추출)**
스키마·쿼리 이력만으로는 비슷한 테이블을 구분 못 한다고 보고, **Codex로 파이프라인 코드를 크롤링**해 각 테이블이 실제로 무엇을 담는지 추출한다. 이를 ① 테이블 사용 ② 사람 주석 ③ Codex 보강 ④ 조직 지식(Slack/Docs) ⑤ 메모리 ⑥ 런타임의 6계층으로 쌓아 매일 오프라인으로 임베딩 → RAG로 검색한다. 코드 기반 맥락이 **자동 갱신**되는 게 핵심. 검증은 Evals API(골든 SQL), 권한은 기존 ACL pass-through.

**Meta — "각자의 도메인은 좁다" (개인화 + 커뮤니티)**
"쿼리의 88%가 최근 90일 내 본 테이블"이라는 검증 가능한 가설에서 출발. 수백만 테이블이 있어도 개인의 작업 영역은 수십 개뿐이라, 문제를 풀 수 있는 크기로 좁힌다. **개인 쿼리 이력을 LLM이 요약한 "shared memory"**가 킬러 피처(동료의 이력을 가리키면 그 사람을 "복제"도 가능). 팀 단위 지식은 Cookbook(패키지) / Recipe(분석 절차) / Ingredient(데이터의 의미) 3계층으로 분리. 커뮤니티가 4,500+ Recipe를 자체 생산했고, 모든 답에 SQL을 노출해 사용자가 직접 검증한다.

**Anthropic — "정답을 하나로 강제한다" (거버넌스)**
오류를 3대 실패 모드(개념↔엔티티 모호성 / staleness / 검색 실패)로 분해하고, 각 레이어가 어떤 모드를 잡는지 매핑한다. 데이터 파운데이션(소수의 canonical 데이터셋) → 소스 오브 트루스(**시맨틱 레이어를 강제 1순위 경로로**) → 스킬(라우터 + 도메인별 reference doc) → 검증의 4계층. 스킬 문서를 데이터 모델과 **같은 repo/PR에 colocate하고 CI로 강제**해 문서가 썩는 걸 막는다(모델 변경 PR의 90%가 스킬 변경 동반). 검증이 가장 두꺼움: offline eval + ablation + online(adversarial review, provenance footer, 정정 자동 수집).

## 2. 공통점 — 세 회사가 똑같이 도달한 결론

1. **모델이 아니라 컨텍스트에 베팅한다.** 셋 다 핵심 투자는 모델이 아니라 그 주변(거버넌스·문서·메모리·평가)이다.
2. **과거 쿼리를 그냥 검색시키는 건 약하다.** OpenAI는 보조 계층, Meta는 가공해서 개인화 재료로, Anthropic은 ablation으로 "raw 코퍼스 접근은 정확도에 거의 영향 없음"을 직접 입증.
3. **반복 작업을 재사용 단위로 패키징한다.** OpenAI Workflows / Meta Recipe / Anthropic Skill — 이름만 다르고 의도는 같다.
4. **투명성이 곧 제품이다.** 검증 불가능한 정답은 쓸모없다는 인식 → SQL 노출(Meta), 추론·실행 링크(OpenAI), provenance footer(Anthropic).
5. **silent failure(그럴듯하지만 틀린 답)는 아무도 못 막는다.** 셋 다 "탐지 가능성을 높일 뿐 방지는 못 한다"고 인정. 업계 공통 미해결 과제.

## 3. 차이점 — 핵심은 "정답을 어디서 가져오고, 어떻게 안 썩게 하는가"

| 축 | OpenAI | Meta | Anthropic |
|---|---|---|---|
| **정답의 출처** | 코드(자동 추출) | 사람의 행동 + 팀 지식 | 사람이 정의한 canonical |
| **갱신 방식** | 코드 크롤링 자동 갱신 | 쿼리 이력 자동 요약 | PR colocation + CI 강제 |
| **인력 부담** | 낮음(자동화 의존) | 중간(커뮤니티 자율) | 높음(사람이 정의 소유) |
| **개인화 vs 표준화** | 중간(공용 계층 + 개인 메모리) | **개인화**가 출발점 | **표준화**(모두에게 같은 답) |
| **검증 깊이** | Evals(회귀 방지) | 정량 평가 적음, human-in-loop | offline+ablation+online (로드맵 의사결정 도구) |

**한 줄 해석:** OpenAI는 **자동화**(코드가 알아서 의미를 채운다), Meta는 **개인화·확산**(각자 도메인을 좁히고 커뮤니티가 키운다), Anthropic은 **거버넌스**(정답을 하나로 모으고 강제로 안 썩게 한다)에 무게를 둔다. 같은 트레이드오프(정확도 ↔ 유지보수 비용)를 푸는 세 가지 좌표인 셈이다.

> Anthropic은 "현재 모델의 약점을 메우려 과한 인프라를 짓기보다, 모델이 좋아지면 사라질 문제는 기다리는 것도 방법"이라고 명시한다. 어떤 레이어를 직접 만들지 결정할 때 유효한 기준.
